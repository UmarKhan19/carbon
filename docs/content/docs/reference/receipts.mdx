---
title: Receipts
description: "Goods coming in: receiving against a purchase order and posting to inventory."
---

A **receipt** brings goods in. Like <Term id="shipment">shipments</Term>, one receipt model serves several sources: most often a <Term id="purchase-order">purchase order</Term>, but also inbound transfers. <Term id="posting">Posting</Term> is what turns a delivery into stock on hand.

A receipt line soft-links back to the purchase order line it fulfills. Posting adds the quantity to inventory, raises the order line's received quantity, and advances the order. When accounting is enabled, it also debits inventory (or <Term id="wip">work-in-process</Term>, for outside processing) against a <Term id="gr-ir">goods-received-not-invoiced</Term> accrual that the supplier invoice later clears.

## Fields

<FieldTable>
  <Field name="Source document" type="reference">The purchase order (or transfer) being received.</Field>
  <Field name="Received quantity" type="number">Units on this receipt, per line.</Field>
  <Field name="Conversion factor" type="number">Converts the supplier's purchase unit to the stock unit.</Field>
  <Field name="Location" type="reference">Where the goods land.</Field>
</FieldTable>

## Status

<StatusFlow entity="receipt">
  <Status name="Draft">Being prepared; nothing posted.</Status>
  <Status name="Pending" branch>Queued to post.</Status>
  <Status name="Posted" accent>Inventory increased and the order advanced.</Status>
  <Status name="Voided" terminal>Reversed after posting.</Status>
</StatusFlow>

<Callout type="info">
Receive an item flagged for inspection and its units land **On Hold**, with an inbound inspection opened at *Pending* — they don't become available stock until the inspection clears.
</Callout>

## Troubleshooting

Exact errors users hit when working with receipts, with cause and resolution.

### "Receipt is empty"
The post dialog blocks posting when no line has a received quantity above zero (and no fixed-asset line is marked received). Enter the received quantity on at least one line, then post again.

### "Batch number is required"
A batch-tracked line has no batch number assigned. Fill in the batch number in the line's batch properties before posting.

### "Serial numbers are missing"
A serial-tracked line has fewer serial numbers entered than its received quantity. Every received unit needs a serial number before the receipt can post.

### "Can only void posted receipts"
Voiding applies only to a receipt at **"Posted"**. A Draft or Pending receipt can be edited or deleted instead — there is nothing to reverse yet.

### "Cannot void a receipt created by a purchase invoice. Void the invoice instead."
This receipt was generated by posting a purchase invoice, so it must be reversed through that invoice: void the purchase invoice and the receipt reverses with it.

### "Cannot delete a posted receipt"
A posted receipt is history and can't be deleted. Void it instead, which writes reversing entries; only Draft and Pending receipts can be deleted.

### "Failed to post receipt"
The generic wrapper when the posting function fails. Most common underlying causes: the [accounting period](/docs/reference/accounting) for the posting date is closed or locked, or the company's default accounts (inventory, goods received not invoiced) aren't configured — the underlying message is then "Error getting account defaults". Check the period first, then the default accounts, and ask the user for the full error text if they have it.
